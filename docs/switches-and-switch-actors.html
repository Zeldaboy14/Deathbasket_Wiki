Switch Basics

What is a switch and what does it do?

Every scene file in the game gets a certain amount of memory in which to store switches or 'flags' (with the exception of Inverted Stone Tower Temple, whose switches are shared with Stone Tower Temple). A switch is simply a single bit that is either on or off (1 or 0) and it allows an actor to store its state between rooms or loading zones. Understanding switches is key to designing dungeons that rely on them. There are four types of switches: generic switches, chest switches, clear switches, and collectible switches. In most cases an actor will define which switch it attaches to through its variable, though there are some cases where its x, y, or z rotation can also be used. While global switches do exist, this article will only focus on switches at the per-scene level. Note that unless stated otherwise, any memory addresses given are for the US N64 version of Majora's Mask.

Brief note: using actors in Majora's Mask

When reading actors from a map/scene file in Majora's Mask, it's important to note that the format in which they are stored is not the same one used by Ocarina of Time, though it is very similar. The major difference is that rotation values in MM are not stored as 16-bit integers but as 9-bit integers that represent the actor's rotation in degrees from 0-359 (wraps around for numbers higher than this). This is the upper 9 bits of what would usually be actor rotation in OoT, with the lower 7 bits being used as an additional variable of some sort. Note: In this article I will hereafter be referring to these as ex. y-rotation and y-variable. The number is stored thus
((rotation & 0x1FF) << 7) | (var & 0x7F)
The x- and z-variables are generally set to 0x07 and 0x7F respectively, while the y-variable is used to attach some actors to cutscenes that will play on activating or deactivating a switch they attach to. Some actors, such as chests, can use these numbers differently. Additionally, some actors can use the upper 4 bits of their actor number to alter their behaviour.

Switch Types

Generic switches

This type of switch is arguably the most important and can be accessed by a great number of the game's actors. There is a total of 128 generic switches and they are used most frequently in dungeons, having three sub-types that depend on the switch number:
000 - 063: semi-permanent
064 - 095: scene-temporary
096 - 127: room-temporary
Semi-permanent switches keep their state until playing the Song of Time or being changed by an actor. It is possible to store switches permanently and this will be addressed later. In dungeons, switch 62 is used by the boss door if there is one.
Scene-temporary switches will be cleared upon leaving the area or playing the Song of Soaring. They are not cleared by voiding out even if this reloads the scene file (ex. drowning as goron).
Room-temporary switches are cleared every time the current room changes and by voids that reload the scene. It is not advisable to use the same room-temporary switches in adjacent rooms as actors may be triggered by the switch before it is cleared. Switch 127 is often used by actors that do not intend to be attached to a switch.

Chest switches

Chest switches are used to keep track of which chests have been opened within the current scene. There is a total of 32 chest switches available and all will keep their state until changed by playing the Song of Time or by an actor (i.e. they are semi-permanent). If multiple chests use the same chest switch then they will all open at the same time, so this should be avoided unless it is necessary.

Clear switches

Clear switches are used per-room (up to 32 rooms) and their value is set when all enemies in a room are defeated. Although there are actually 64 clear switches, the first 32 are semi-permanent and the second 32 are room-temporary. The way this works is that for each room, the room-temporary clear switch is set when no enemies are active but the corresponding semi-permanent switch is set iff an actor in the room is attached to the clear switch and no enemies are active. If the semi-permanent switch is set for any room then enemies will no longer spawn upon entering that room. As before, the room-temporary switches are cleared upon leaving the room and enemies will spawn again when going back to it.

Collectible swtiches

These switches are set when certain items are picked up, such as heart pieces or rupees. There is a total of 64 collectible switches, where the first 32 are semi-permanent and the second 32 are scene-temporary.

Reading Switches

For the US N64 version of Majora's Mask, the current scene's switches are stored at 0x803E8978 in RAM. They are laid out as follows:
803E8978 - 128 bits: generic switches
803E8988 - 032 bits: chest switches
803E898C - 064 bits: clear switches
803E8994 - 064 bits: collectible switches

Switches are stored in 32-bit blocks and read starting from the least significant bit of each. For example, if generic switches 31 and 34 were set, the data would look like:
803E8978: 80000000 00000004 00000000 00000000
i.e. the first word stores switches 0-31, the second stores switches 32-63, etc.

Using built-in functions

Of course, actors themselves are not programmed to read or write switches directly from the aforementioned memory addresses. Instead, the game provides simple functions for this purpose:

/* generic switch functions */
0x800B5BB0: int  get_generic_switch(void *mmctxt, int switch_number) /* returns non-zero if switch is set */
0x800B5BF4: void set_generic_switch(void *mmctxt, int switch_number) /* sets switch as on */
0x800B5C34: void remove_generic_switch(void *mmctxt, int switch_number) /* sets switch as off */

/* chest switch functions */
0x800B5C78: int  get_chest_switch(void *mmctxt, int chest_number) /* returns non-zero if chest is open */
0x800B5C90: void set_chest_switch(void *mmctxt, int chest_number) /* sets chest as opened */

/* clear switch functions */
0x800B5CC4: int  get_clear_switch(void *mmctxt, int room_number) /* returns non-zero if room is clear */
0x800B5CDC: void set_clear_switch(void *mmctxt, int room_number) /* sets room as clear (permanent) */
0x800B5CF8: void remove_clear_switch(void *mmctxt, int room_number) /* removes room's clear status */
0x800B5D18: int  get_temp_clear_switch(void *mmctxt, int room_number) /* returns non-zero if room is clear (temp only) */
0x800B5D30: void set_temp_clear_switch(void *mmctxt, int room_number) /* sets room as clear (temp only) */
0x800B5D4C: void remove_temp_clear_switch(void *mmctxt, int room_number) /* removes room's temp clear status */

/* collectible switch functions */
0x800B5D6C: int  get_collectible_switch(void *mmctxt, int switch_number) /* returns non-zero if switch is set */
0x800B5DB0: void set_collectible_switch(void *mmctxt, int switch_number) /* sets collectible as obtained */
Here mmctxt is the so-called 'global context', which is address 0x803E6B20. This is used as a first argument to many of the game's functions and some important variables are stored nearby. In most cases actors that use the above functions will pull switch numbers from their variables, but some, such as some actors in Great Bay Temple, have the switch numbers hardcoded into their functions in order to attach to multiple switches at once. This is also seen in some scene-specific code the game uses.

Making switches permanent

Upon leaving an area, all semi-permanent switches for that scene are moved to another location in RAM to be stored until they are needed again or until the Song of Time is played. Of these, the generic, chest, and collectible switches are able to be stored permanently in the game save so that they will not be reset by playing Song of Time. To do this, the game applies a bit mask to the switches in their current state to clear any that are not to be saved permanently. The table of bit masks for permanent flags starts at 0x801C5FC0 in RAM and 0xC5C500 in ROM (decompressed) and contains 16 bytes of data for each scene:
32 bits: mask for generic switches 00-31
32 bits: mask for generic switches 32-63
32 bits: mask for chest switches
32 bits: mask for collectible switches 00-31

Masks are applied to switches in the following way:
permanent_switches = switches & mask

Note that any permanent switches can still be deactivated later if an actor can attach to and disable them. 'Permanent' in this sense refers to their ability to be carried across cycles without being reset. Curiously, the only scene for which the mask for chest switches is non-zero is the Laundry Pool, though no chest is present there in the game.

Switch cutscenes

If an actor is attached to a switch then its y-variable can also be used to attach it to a cutscene that will play when the switch is activated. A value of 0x7F is used when no cutscene should play. It is recommended that you know the basics of reading a scene file to understand this section.

Finding and reading cutscene definitions

The starting point here is to find the 0x1B command in the scene file's header. This command states the number of cutscene definitions and their start address within the file. At this address there is 16 bytes of data per cutscene definition in the following format, and the y-variable of an actor is used as an index for this list:

struct cs_def {
int16 unk1;
int16 frame_count;
int16 camera_index;
int16 cutscene_number;
int16 unk2;
uint8 sound_effect;
uint8 unk3;
int16 hud_fade;
uint8 return_camera;
uint8 letterbox;
};

In more detail:
frame_count is the number of frames over which the game will be in cutscene mode. After this number of frames has elapsed, control of Link and the camera is regained. Setting this value to -1 appears to have the game calculate an appropriate frame count instead of defining it manually.
camera_index defines the position in the scene's camera list from which to get camera data. This will be elaborated on in the next section.
cutscene_number overrides camera_index and plays a predefined cutscene from the scene file. This will be ignored if equal to -1.
sound_effect sets which sound (puzzle complete noise) should be played when the cutscene finishes. Using 0 plays no sound.
hud_fade determines the HUD behaviour during the cutscene. Using 0 makes it disappear, 1 does nothing, and 2 leaves the A button on screen.
return_camera defines how the camera returns to Link after the cutscene. A value of 0 makes it pan back to Link and 1 makes it instantly return.
letterbox sets the size of the black bars to be drawn at the top and bottom of the screen.
Unknowns require more research.

For this article, camera_index will be the most important number here. The data used by cutscene_number is complicated and will not be explained.

Finding and reading camera data

The camera list is defined by the 0x02 command in the scene header. Again, it defines the size of the list and its address. Each entry is made up of two 16-bit unknowns and a 32-bit pointer to the camera data for that entry. To find the relevant pointer for a value of camera_index:
pointer_address = camera_list_start + (camera_index << 3) + 4
This 32-bit value at this address will point to the camera data itself, which is made up of 18 bytes:

struct camera_data {
int16 x_pos;
int16 y_pos;
int16 z_pos;
int16 x_rot;
int16 y_rot;
int16 z_rot;
int16 fov;
int16 pan_speed;
int16 unk; /* seems to always be -1 */
};

This is quite simple:
x_pos is the x position the camera will move to.
y_pos is the y position the camera will move to.
z_pos is the z position the camera will move to.
x_rot is the camera's rotation around the x-axis.
y_rot is the camera's rotation around the y-axis.
z_rot is the camera's rotation around the z-axis.
fov is the field of view in degrees.
pan_speed is how quickly the camera will translate/rotate to the specified position/rotation.
